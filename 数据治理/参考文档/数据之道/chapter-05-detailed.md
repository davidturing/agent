# 5. 面向“联接共享”的数据底座建设

_在从信息化向数字化转型的过程中，企业积累了海量数据，但这些数据普遍存在分散、不拉通的问题，缺乏统一的定义和架构，导致真正能产生价值的数据很少，找到想要、能用的数据越来越难。_ 本章将详述华为通过构建**数据底座**，实现数据的汇聚和联接，打破数据孤岛和垄断，重建数据获取方式和次序，从而在数字化转型中起着关键作用。

## 5.1 支撑非数字原生企业数字化转型的数据底座建设框架

**数据底座** (Data Foundation) 是华为提出的一个核心概念，其目标是通过将公司内外部的数据汇聚在一起，对数据进行重新组织和联接，让数据拥有清晰的定义和统一的结构，并在此基础上，让数据更容易获取，最终打破数据孤岛和垄断。

**数据底座**的核心目标可以概括为四点：
1.  **统一管理**：统一管理结构化、非结构化数据，将数据视为资产，能够追溯数据的产生者、业务源头以及需求方和消费者。
2.  **打通数据供应通道**：为数据消费者提供丰富的数据原材料、半成品以及成品，满足自肋分析、数字化运营等不同场景的数据消费需求。
3.  **确保数据完整、一致、共享**：监控数据全链路下的各个环节的数据情况，从底层数据存储的角度，诊断数据冗余、重复以及“僵尸户”问题，降低数据维护和使用成本。
4.  **保障数据安全可控**：基于数据安全管理策略，利用数据权限控制，通过数据服务封装等技术手段，实现对涉密数据和隐私数据的合法、合规消费。

### 5.1.1 数据底座的总体架构

华为的**数据底座**由**数据湖**和**数据主题联接**两层组成，将公司内外部的数据汇聚到一起，并对数据进行重新组织和联接，为业务可视化、分析、决策等提供数据服务。其总体架构（基于图5-1复现）如下：

1.  **数据源 (Data Source)**
    *   这是架构的最底层，是所有数据的起点。
    *   **内部数据**：涵盖企业运营的各个核心流程系统，如 LTC (Lead to Cash)、ITR (Issue to Resolution)、SD (System Delivery)、HR、IPD (Integrated Product Development)、FIN (Finance)、PRO (Procurement)、MFG (Manufacturing)、SCM (Supply Chain Management) 等。
    *   **外部数据**：来自企业外部的第三方数据。

2.  **数据底座 (Data Foundation)**
    *   **数据湖 (Data Lake)**：实现企业数据的“**逻辑汇聚**”。数据通过标准化的入湖流程进入数据湖。
        *   **入湖标准**：数据入湖必须遵循六项标准，包括明确**数据Owner**、发布数据标准、定义数据密级、认证数据源、数据质量评估、元数据注册。
        *   **入湖方式**：分为**物理入湖**（数据复制到湖中）和**虚拟入湖**（通过虚拟化技术访问源数据）。
    *   **数据主题联接 (Data Subject Connection)**：将**数据湖**中的原始数据按业务流/事件/对象/主题进行联接和处理，形成面向数据消费的主题数据。
        *   **联接方式**：主要有五种数据联接方式，包括**多维模型**、**图模型**、**指标**、**标签**和**算法模型**。
        *   **主题域划分**：数据被组织成不同的主题，如产品主题、合同主题等，形成虚拟的、面向分析的**主题表 (Fact/Dimension Tables)**。

3.  **数据服务 (Data Services)**
    *   这是**数据底座**的价值出口，将处理好的数据提供给上层应用。
    *   **消费场景**：支持报告、自助分析、实时可视、智能决策、Data Lab（数据实验室）等多种数据消费场景。

4.  **数据资产管理 (Data Asset Management)**
    *   这是一个贯穿所有层级的管理体系，确保**数据底座**的健康运行。
    *   **核心能力**：包括元数据管理、指标管理、数据血缘、数据管控、数据质量和数据安全。

### 5.1.2 数据底座的建设策略

**数据底座**建设不能一蹴而就，要从业务出发，因势利导，持续进行。华为**数据底座**建设采取**“统筹推动、以用促建、急用先行”**的建设策略，并遵循以下四项基本原则：

1.  **数据安全原则**
    *   **数据底座**的数据资产应遵循用户权限、数据密级、隐私级别等管理要求，以确保数据在存储、传输、消费等全过程中的数据安全。
    *   _技术手段包括但不限于授权管理、权限控制、数据加密、数据脱敏。_

2.  **需求、规划双轮驱动原则**
    *   **数据底座**的数据资产建设基于业务规则和需求来触发，采用双轮驱动的原则进行建设，对核心数据资产优先建设。

3.  **数据供应多场景原则**
    *   **数据底座**的数据资产需根据业务需求提供离线/实时、物理/虚拟等不同的数据供应通道，满足不同的数据消费场景。

4.  **信息架构遵从原则**
    *   **数据底座**的数据资产应遵从公司的信息架构，必须经 IA-SAG (信息架构专家组) 发布并完成注册。

## 5.2 数据湖：实现企业数据的“逻辑汇聚”

**数据湖**是逻辑上各种原始数据的集合，除了“原始”这一特征外，还具有“海量”和“多样”（包含结构化、非结构化数据）的特征。**数据湖**保留数据的原始格式，原则上不对数据进行清洗、加工，但对于数据资产多源异构的场景需要整合处理，并进行数据资产注册。

### 5.2.1 华为数据湖的3个特点

1.  **逻辑统一**
    *   华为**数据湖**不是一个单一的物理存储，而是根据数据类型、业务区域等由多个不同的物理存储构成，并通过统一的元数据语义层进行定义、拉通和管理。

2.  **类型多样**
    *   **数据湖**存放所有不同类型的数据，包括企业内部IT系统产生的结构化数据、业务交易和内部管理的非结构化文本数据、公司内部园区各种传感器检测到的设备运行数据，以及外部的媒体数据等。

3.  **原始记录**
    *   华为**数据湖**是对原始数据的汇聚，不对数据做任何的转换、清洗、加工等处理，保留数据最原始特征，为数据的加工和消费提供丰富的可能。

### 5.2.2 数据入湖的6个标准

数据入湖是数据消费的基础，需要严格满足入湖的6项标准，包括明确**数据Owner**、发布数据标准、定义数据密级、明确数据源、数据质量评估、元数据注册。通过这6项标准保证入湖的数据都有明确的业务责任人，各项数据都可理解，同时都能在相应的信息安全保障下进行消费。

1.  **明确数据Owner**
    *   **数据Owner**由数据产生对应的流程Owner担任，是所辖数据端到端管理的责任人，负责对入湖的数据定义数据标准和密级，承接数据消费中数据质量问题，并制定数据工作路标，持续提升数据质量。

2.  **发布数据标准**
    *   入湖数据要有相应的业务数据标准。业务数据标准描述公司层面需共同遵守的“属性层”数据的含义和业务规则，是公司层面对某个数据的共同理解。

3.  **认证数据源**
    *   通过认证数据源，能够确保数据从正确的源头入湖。认证数据源应遵循公司数据源管理的要求，一般数据源是指业务上首次正式发布某项数据的应用系统，并经过数据管理专业组织认证。

4.  **定义数据密级**
    *   定义数据密级是数据入湖的必要条件，为了确保数据湖中的数据能充分地共享，同时又不发生信息安全问题。入湖的数据必须须要定密。

5.  **数据质量评估**
    *   数据质量是数据消费结果的保证，数据入湖不需要对数据进行清洗，但需要对数据质量进行评估，让数据的消费人员了解数据的质量情况，并了解消费该数据质量的质量风险。

6.  **元数据注册**
    *   元数据注册是指将入湖数据的业务元数据和技术元数据进行关联，包括逻辑实体与物理表的对应关系、业务属性和表字段的对应关系。通过关联业务元数据和技术元数据的关系，能够支撑数据消费人员通过业务语义快速地搜索到数据湖中的数据，降低数据湖消费的门槛，让更多地业务分析人员理解和消费数据。

### 5.2.3 数据入湖方式

数据入湖遵循华为信息架构，以**逻辑实体**为粒度入湖，原则上一个逻辑实体的所有属性应该一次性进湖，避免一个逻辑实体多次入湖，增加入湖工作量。

数据入湖的方式主要有**物理入湖**和**虚拟入湖**两种，根据数据消费的场景和需求，一个逻辑实体可以有不同的入湖方式。

**五种主要技术手段对比**：

| 入湖方式 | 说明 | 数据搬家 | 实时性 | 源系统性能要求 | 批量数据处理 | 历史数据处理 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **物理入湖** | **批量集成** | ETL/ELT工具 | 需要 | 非实时 | 低 | 支持 | 支持 |
| | **数据复制同步** | CDC工具 | 需要 | 实时 | 中 | 不支持 | 不支持 |
| | **消息集成** | MQ工具 | 需要 | 实时 | 中 | 不支持 | 不支持 |
| | **流集成** | Pipeline工具 | 需要 | 实时 | 中 | 支持 | 支持 |
| **虚拟入湖** | **数据虚拟化** | 虚拟化工具 | 不需要 | 实时 | 高 | 支持(弱) | 支持(弱) |

**数据湖的主动与被动入湖方式**：

| 入湖方式 | 数据源 | 数据湖 |
| :--- | :--- | :--- |
| **PULL (拉)** | 被动：当被请求时提供数据 | 主动：决定何时获取数据 |
| **PUSH (推)** | 主动：按自己节奏提供数据 | 被动：响应接收数据 |

### 5.2.4 结构化数据入湖

结构化数据是指由二维表结构来逻辑表达和实现的数据，严格遵循数据格式与长度规范，主要通过关系型数据库进行存储和管理。其入湖流程（基于图5-3复现）如下：

1.  **数据入湖需求分析及管理**
    *   **规划驱动**：由业务Owner或数据代表发起，输出入湖规划清单。
    *   **需求驱动**：由数据消费者发起，输出入湖需求。

2.  **检查数据入湖条件和评估入湖标准**
    *   **检查数据源准备度**：IT团队提供源系统的物理表规范度，数据代表评估源系统的数据质量。
    *   **评估入湖标准**：检查是否满足明确Owner、发布标准、认证源、定义密级等6大标准。

3.  **实施数据入湖**
    *   数据代表依据消费场景合理选择入湖方式（物理或虚拟），并负责实施。

4.  **注册元数据**
    *   元数据是公司的重要资产，是数据共享和消费的前提。入湖后，数据代表需检查并注册元数据。

### 5.2.5 非结构化数据入湖

非结构化数据包括无格式的文本、各类格式的文档、图像、音频、视频等多样异构的格式文件。其管理不仅包括文件本身，而且包括对文件描述的元数据信息。

**非结构化数据入湖的4种方式**：

1.  **基本特征元数据入湖**：仅将文件的基本元数据（如文件名、创建者、密级等）存入数据湖，文件本身仍在源系统。
2.  **文件解析内容入湖**：对源文件的文件内容进行文本解析、拆分后入湖。
3.  **文件关系入湖**：从源端提取的文件上下文关系入湖，如知识图谱应用。
4.  **原始文件入湖**：根据消费应用案例从源端把原始文件搬入湖。

_参考都柏林核心元数据，非结构化数据的基本特征类属性元数据规范如：数据(Code)、是否必填(Title)、类型(Type)、格式(Format)、创建者(Creator)、主题(Subject)等15个核心元数据。_

## 5.3 数据主题联接：将数据转换为“信息”

在数字化转型的背景下，数据消费已经不再局限于传统的报表分析，还需要支持用户的自助分析、实时分析，通过数据的关联，支持业务的关联影响分析以及对目标对象做特征识别，进行特定业务范围圈定、差异化管理与决策等。

### 5.3.1 5类数据主题联接的应用场景

华为在数据湖的基础上通过建立数据联接层，基于不同的分析场景，通过5类联接方式将跨域的数据联接起来，将数据由“原材料”加工成“半成品”和“成品”，支撑不同场景的数据消费需求。

1.  **多维模型**
    *   **场景**：用于多角度分析，如区域、产品、客户等多维度的业务关系分析。
    *   **作用**：建立基于事实表、维度表以及相互间联接关系，实现多维数据查询和分析。

2.  **图模型**
    *   **场景**：面向关系的关联分析，如查看某国家产地的数据具体关联到哪个客户以及合同、订单的详细信息。
    *   **作用**：通过建立数据对象以及数据实例之间的关系，帮助业务快速定位关联影响。

3.  **标签**
    *   **场景**：对特定业务范围的圈定，如用户画像、产品营销策略支持。
    *   **作用**：在业务场景的上下文背景中，运用抽象、归纳、推理等算法计算并生成目标对象特征的表示符号。

4.  **指标**
    *   **场景**：对业务结果、效率和质量的度量，如促销员门店覆盖率指标。
    *   **作用**：通过数据计算得到衡量目标总体特征的统计数值，能够客观地表征企业某一业务活动中业务状况。

5.  **算法模型**
    *   **场景**：面向智能分析的场景，如预测未来18个月的销售量。
    *   **作用**：通过数学建模对现实世界进行抽象、模拟和仿真，提供支撑业务判断和决策的高级分析方法。

### 5.3.2 多维模型设计

**多
